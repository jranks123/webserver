Project

Introduction

For our Web Tech coursework we have acquired and completed a real world client request. We have undertaken the responsibility of creating a website and content management system for the band Arcade Fire. The website should be aesthetically stunning, in tune with the band's latest album campaign, and very easy to navigate still positioning itself as a website suitable for 2014. The band's fanbase is split between the tech-savvy and the tech-illiterate, so we aim to cater for this at all times.

During this coursework we will be communicating with the band and the band's management for their approval on stylistic choices and all final assets included are approved for use by the band, or the respective owners of the media.

Client

Arcade Fire is a Grammy winning indie rock band with four platinum albums. The band's heavy usage of beautiful and stunning imagery requires the design of the website to be bordering on artistic, while making sure the navigation is still simple and unconfusing. The band has provided a variety of exclusive content for us to utilize in the design of the website, including 150 unseen images in the form of a digital booklet which we have manipulated and edited to create the core aesthetics of our finished site.

The band have also requested that the site be updateable by them (biography, tour dates, etc) and have longevity enough to exist for the remainder of their current album campaign, Reflektor, which could last until 2017.

Client Side

Basics

We have attempted to create a fluid one-page scrolling website that is engaging and easy to navigate for a fan of Arcade Fire. By combining HTML with CSS and JavaScript we were able to create a dynamic, responsive design that immerses the user while remaining easy to navigate at all times.

Since Arcade Fire are an extremely large, image-orientated indie rock band it was important for our website to be aesthetically stunning and in matching with the band's visual branding without overloading a user with images and spoiling the user-experience.

In addition to creating the standalone website for the assignment, Arcade Fire requested we also skin a blog/news style website to act as both a temporary website during the development process of the main site and later retire to act as the blog section of the website. By rebranding and skinning a WordPress install for the blog section we allowed the band members to remain on a platform they are comfortable using, while still providing a website that is visually appealing to users, and in theme with the main website.

This blog is already live on the client's website viewable here. We will discuss the work and process for this in a later section of this report.

As a brief summary, during this project we have used: XHTML, CSS, JavaScript, jQuery, PHP (minor for WordPress implementation), Photoshop, Gimp, Inkscape and attempted animation with SVGs, JavaScript and CSS. We have attempted to use as few CSS libraries as possible to focus on implementing things such as responsive design by hand.

We had 2 CSS errors in main.css, however they are vital to the CSS animation of our page load element and for this reason we have left them in our CSS as requested by our client. This animation seems to work across all major browsers and seemed like CSS animation was a better option than using a .gif. We are unsure if this will reduce our mark so have made a note here that we are aware of it and have opted to keep it in.

Navigation

One of the first key elements of our website was the navigation bar. We wanted to create a navigation interface that existed below the page's initial introduction header, but became 'sticky' to the page top before it is able to exit the screen.

After laying out the navigation bar in HTML and CSS, we used JavaScript and an additional CSS class to achieve this effect. We added a 'div' element to mark the point where the navigation bar should acquire or release it's 'sticky' property and used the distance to scroll to this div as the deciding factor as to whether or not the navigation bar should be pinned.

To pin the navigation we simply added a class named 'navbar-fixed' to the navbar which changed it's position to fixed at the top of the window, and the z-index to a high number allowing it to exist in front of all other elements of the website, freeing the navigation for use at all times.

Navigation's Responsive Design

From the beginning of the assignment we attempted to keep our site's design responsive. This began with the navigation bar. When the screen size reached that of a tablet or mobile phone, we changed the style of our navigation bar to become more symmetrical and adopt the aesthetic of a phone app. See below:

This symmetrical style makes better use of webpage real-estate and is intended to be more intuitive to a mobile user. In modifying the CSS of the navigation bar, particularly the bar's height, we also had to modify the scroll-points for the page as a larger nav bar with a high z-index would cover up content.

Additional cosmetics and minutia were taken into consideration with the navigation bar. Namely, when the navigation bar is resting over the site header it's opactiy is decreased for aesthetic purposes. However, when it is over the website's content the opacity increases as to make sure the layout is unconfusing to a user. Furthermore, opacity increases again in the mobile layouts to act as an app navigation bar and as less of a style element. Finally, since old versions of Internet Explorer struggle with rgba values we included a regular rgb fall-back css value for the navigation bar.

Further Responsive Design

We attempted to make all elements of the site's design responsive to browser changes, including the album artwork resizing, columns realigning and sections re-designed for different sizes. We used % where possible for major elements, pixels for specific minor positioning and em when elements were related to fonts.

Animated Header Logo (.svg)

We created a large svg version of Arcade Fire's logo as decoration to exist above the main header image using Inkscape. Using the opacity channel we were able to create the illusion of a steady fade upon scroll down, and used real SVG animation upon page load (this works in Chrome, but unsupported in Firefox).

Parallax Elements

Another style choice we made during the website creation was the use of parallax background elements to provide a sense of depth for a user while scrolling. Having some experience with parallax headers prior to this assignment we were able to write this code from memory, though originally learning how to create these effects through multiple tutorials online.

Our exact implementation requires, in CSS, the background of a div to be held in position: fixed and centered in the x-direction, then upon scroll jQuery (see implementation in parascroll.js) calculates the background-position in the y-direction that the image is modified, with respect to the scrolling rate.

Glide Scrolling

To accompany the parallax backgrounds we implemented javascript 'glide scrolling', a technique to automatically scroll a user to a specific point in the page in a clean and quick way. We have both had experience implementing this previously, after learning how it works in previous web self-study, so for this assignment we implemented a slightly modified version of a basic smooth slider we had implemented months ago (link) and adapted the animation speed to suit for this project.

This glide scrolling is utilized by the navigation bar, the footer navigation bar, the 'return to top' image at the bottom of the page and each section's footer logo which acts as a 'move-to-next-section' button.

Music Hidden Drawer

We have implemented a hidden drawer in CSS and JavaScript for the Music section of the website. By animating the div and increasing the height we are able to create space for content to to extend on the option a user interacted with. The content divs for each option are hidden off the side of the page with height=0, when an option is pressed the jQuery checks to see if a content div currently occupies the space. If not, the option's selected respective content animates in from the side. Otherwise, the content currently occupying this area leaves and is replaced by a new div. This was a custom jQuery implementation tailored for this site's content and was by far the most time consuming aspect of the project as attempting to keep this section's design responsive was a lengthy and troublesome task.

When this drawer opens it displays extended information about the album, as well as a mockup of a music player and some buy links. We created the mockup music player in Photoshop, basing it on the Spotify app's styling. By creating a template of music player on one layer, with a secondary layer below for album artwork we were able to quickly create multiple music player mockups with the same dimensions and aesthetic. However, in the server side coursework we aim to implement a real custom music player.

Additionally in this section, we created animated windows on hover of the album artworks. This is a largely modified implementation of this tutorial which we used to learn more about CSS animation.

'Loading...'

We created a preloader that allows the site to have an air of professionalism while it loads. Instead of witnessing the images steadily load in front of a user, a loading catch page makes sure the site is not displayed until it is complete. Neither of us had experience with creating pre-loaders before, however we had attempted to implement lightboxes in a previous project and applying the same principle was able to create one quickly.

Using a fixed div that occupies the full height and width of the screen, containing only a CSS-animated rotating .png created in Photoshop, creates the illusion of a loading page while hiding the real page-load behind it. We use jQuery to notice when the page has finished loading and then fade out the elements of the the preloader. The jQuery fade out animations were modified from an old lightbox implementation, which came from a tutorial used for self-study last year.

Circular Image Gallery

We implemented a simple image flow gallery where a single image changes to multiple different source images by animation. This was done following the tutorial of Paul Mason as an exercise of using transparent area of images for decoration.

While the jQuery is simple for this gallery, we added an extra effect by creating all of the images for the gallery in Photoshop. We created a template size and included all of the images on seperate layers, before placing a circle object over the top of them. By selecting the outline of the circle and then selecting inverse, before returning to an image layer and deleting the 'outside' of the circle it made it appear as though all the images are peering through a circular window due to their .png transparency.

Cross-browser compatibility

Our site works perfectly as required in Chrome, Firefox and Internet Explorer 9 after including some minor CSS fall-backs to cater for the differences between these browsers. We have also catered for previous versions of Internet Explorer by including fall-back CSS stylesheets tailored to them - these particularly focus on the browser's lack of support for transparency and opacity effects.

Admin Panel & AngularJS

We created a one-page-app admin panel using AngularJS. It had to be easy to navigate and only available to approved administrators. The layout of the admin panel is very simple in terms of HTML and CSS, and the functionality is partly based on AngularJS which injects content partials client-side for each editable section into the admin panel homepage. This approach allows the app to be scalable for future development if further sections of the site required editing, and requires no page reloading when navigating the admin panel.

Reduced page loading for administrators is a small touch based on a request from the band in which they noted that due to their traveling schedule they are often in places with expensive data, or poor data connections, so removing the burden of reloading the same information multiple times is useful, even though quite insignificant in scope due to the lightweightness of the app's admin panel.

Server Side

NodeJS

For the server we decided to use NodeJS, as this is a very new technology we were mostly unfamiliar with, and we were eager to learn it. Since our Content Management System application was due to be quite lightweight in terms of server-side computation, NodeJS allowed us to create the whole thing in Javascript, even including the database for which we were able to use JSON object storage in the form of MongoDB.

NodeJS is less useful for relational database applications, however since this was not a requirement in our CMS specification, it was not a problem.

MongoDB

We used MongoDB as our database for the website. MongoDB has a very unrestricted data model and is great for lots of small data operations while being highly scalable, which is a perfect fit for our lightweight CMS. It is also useful for it's performance with high concurrency reads.

Content Management Overview

The core of our application was to provide the ability for the band to edit the content on their website without requiring them to dig into the code of the site. To do this we created a database to hold the website's current content, used EJS templating to display this database's on the website in the relevant sections, and created an admin panel backend to allow an administritive user to modify what is stored in this database (and thus displayed on the main site).

This implementation means any member of the band could modify the content on the website from an easy-to-use control panel with no programming experience and very little tech-savvy.

The admin panel allows the user to:

  • - Edit band biography
  • - Update all 5 albums' information and buy links
  • - Change the website's featured videos
  • - Add new live dates
  • - Remove erroneous live dates
  • - Change their admin password
  • - Create a new administrator account

Authenticated Access

It was important to make sure that only an administrator is able to edit the website. To do this, we implemented authenticated access for the admin panel. We implemented a 'max_age' on a user's session to allow the maximum time of an admin panel session to be 5 minutes. While this would not be a typical implementation and could even perhaps become irritating for a different type of application (a social network for example), the client specifically requested auto-logout functionality after a set time. This request was due to previous events where the band had left important accounts with millions of followers/subscribers logged in on public computers during their travels/use of internet cafes/borrowed laptops. This feature now acts as a tailored log-out failsafe for this specific client, however would be reconsidered if this CMS were to be extended and repurposed for more general use.

Authenticated user documents were stored in our database as a username, email address, user ID and hash (which was the password stored as a SHA1 hash with salt to improve security and make sure user's passwords were not stored as plaintext). To verify a users password we simply hash the potential password with the same hashing process and check against that users hash value in the database. We integrated Passport.js to handle the authenticate requests.

To improve security, every time a new password was created (either changed by user, or in the instance of a new admin account being created) a new salt was used, 8 characters long and created randomly.

Data Validation (Both Client & Server Side)

From within the admin panel the band can now update the various sections of their website without interacting with the html/ejs. For this reason it is important to do various checks to ensure the data they are entering in the website is valid and won't break the website.

We implemented both client side validation and server side validation. Implementing client side validation allowed us to easily alert the user to any mistakes they may have made when entering data without sending the data to the server. This is useful for regular site users and honest mistakes. However, it is easy to bypass client side form validation so it was also necessary for us to implement server side validation to prevent incorrect data ending up in the database on the chance the client side was bypassed, either accidentally or maliciously.

Here are some examples of the variety of validation implemented:

Simple 'is not empty' validation. This was checked server side using req.assert's notEmpty() capability. Client side check was implemented using form validation, checking the form's value was not simply null or an empty string.

URL validation to make sure ticket links provided by the band or management were in the form of real URLs that linked to external sites (must begin with http:// or https://) and contains no trailing whitespace.

YouTube ID validation to verify the ID entered is between 10 and 12 characters and is strictly alphanumeric.

Validation that both passwords match, and are longer than 8 characters when creating a new password.

Parsing New Content & Data

Having received new information from the administrator, it is sometimes necessary to prepare this data for view or store the data in a special way as it is not always in the format required for display. Examples of this include the 'Edit Biography' option of the admin panel, and the YouTube ID input.

YouTube ID

The admin has the option of providing new YouTube IDs to embed any YouTube video into the website as featured videos. While this is intended to be Arcade Fire's music videos, this functionality will work for any video on YouTube given the video is public or unlisted, and embeddable on external sites.

This works via injecting only the ID into an already correctly parsed template with EJS. A more interesting use for these IDs was to automatically grab a thumbnail for the video from YouTube for use in our video carousel. These thumbnails were automatically served from YouTube as .jpg and we were unable to receive them as .png but we left them in regardless to maintain streamlined functionality.

https://

To create an HTTPS server, needed to produce an SSL certificate. To do this we used openssl with the genrsa command to produce a key, they used this key to produce a certificate. We then sent the key and the certificate to the https.createServer node module, which created our https server.

Our site will only work over https, and this with a combination of the cookie being set to httpOnly means that our cookies cannot be read locally via JavaScript, a big potential security spot. Our cookie is also set to secure, which means that the cookie can no longer be sent with a request that isn’t made over the HTTPS scheme, which further makes our site more secure. If anyone tries to access our site via http, it redirects them to https.

YouTube embed support was affected by using https, forcing us to discover and consult the YouTube API Blog to fix this error (our links were hardcoded as http://... instead of https://...)

X-Site Request Forgery

Connect CSRF allows protection against cross-site request-forgery attacks using middleware tokens placed in a hidden form element and stored in the session. This is works by access to the form being forbidden if the hidden token is not located as present, or more simply if the session token does not match the body token.
